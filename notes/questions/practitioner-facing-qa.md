# 실무자용 AI 개발 Q&A

## 목적

이 문서는 개발자와 개발 조직 실무자가 AI를 활용해 실제 개발을 진행할 때 참고하기 위한 말하기용 Q&A다.

핵심 목적은 AI에게 무작정 "해줘"를 반복하는 방식이 아니라, SPEC, 작업 분해, 검증 루프, 코드 리뷰, 유저 관점 확인을 갖춘 에이전틱 엔지니어링 방식으로 일하도록 돕는 것이다.

## 빠른 요약

```text
1. SPEC을 먼저 만든다
2. AI가 처리 가능한 작은 작업으로 나눈다
3. test / typecheck / lint / e2e / 요구사항 체크 루프를 준비한다
4. AI에게 구현과 반복 개선을 맡긴다
5. 사람은 코드 구조와 영향 범위를 리뷰한다
6. 개발 환경에서 유저 관점으로 동작을 확인한다
7. QA로 넘기기 전 개발자가 최종 책임을 진다
```

## 1. AI로 개발할 때 제일 먼저 해야 하는 건 뭔가?

### 질문

> AI로 개발할 때 제일 먼저 해야 하는 건 뭔가요?

### 답변 프레임

```text
1. 코드를 바로 시키지 않는다.
2. 요구사항을 SPEC과 성공 기준으로 만든다.
3. AI가 처리 가능한 작은 작업으로 쪼갠다.
4. 검증 루프를 만든 뒤 구현을 맡긴다.
```

### 말하기용 답변

AI로 개발할 때 제일 먼저 할 일은 코드를 시키는 것이 아닙니다.

먼저 요구사항을 SPEC으로 만들어야 합니다.
무엇을 만들고, 어떤 조건을 만족해야 하고, 무엇이 되면 완료인지 정해야 합니다.

그 다음 작업을 AI가 처리 가능한 작은 단위로 쪼개야 합니다.
큰 요청을 한 번에 던지면 AI가 국지적으로는 맞지만 전체적으로는 이상한 방향으로 갈 가능성이 커집니다.

그리고 테스트, typecheck, lint, e2e, 브라우저 확인, 요구사항 체크처럼 AI가 실패를 보고 다시 고칠 수 있는 검증 루프를 준비해야 합니다.

### 핵심 한 줄

> AI로 개발할 때 먼저 할 일은 코드를 시키는 것이 아니라, SPEC을 만들고 작업을 쪼개고 검증 루프를 준비하는 것이다.

## 2. SPEC은 어느 정도까지 써야 하나?

### 질문

> SPEC은 어느 정도까지 써야 하나요?

### 답변 프레임

```text
1. 완벽한 기획서까지는 필요 없다.
2. 하지만 성공 기준은 명확해야 한다.
3. 테스트, 화면 확인, API 응답, 권한별 동작처럼 검증 가능한 기준이 있어야 한다.
4. AI가 초안을 만들 수 있지만 최종 SPEC은 사람이 확정해야 한다.
```

### 말하기용 답변

SPEC이 완벽한 기획서일 필요는 없습니다.
진행하면서 발견되고 구체화되는 요구사항도 많습니다.

하지만 AI가 성공과 실패를 구분할 수 있을 만큼은 명확해야 합니다.
무엇이 되면 완료인지, 어떤 조건을 만족해야 하는지, 어떤 동작은 하면 안 되는지 정해야 합니다.

가능하면 검증 가능한 형태가 좋습니다.
테스트 조건, 화면에서 확인할 수 있는 버튼과 상태, API 응답, 권한별 표시 여부, 저장 후 실제 데이터 반영 여부 같은 기준이 있어야 합니다.

AI가 SPEC 초안을 만들고 질문을 던지고 정리할 수는 있습니다.
하지만 최종 성공 기준은 사람이 확정해야 합니다.

### 핵심 한 줄

> SPEC은 완벽한 문서일 필요는 없지만, AI가 성공과 실패를 구분할 수 있을 만큼 검증 가능해야 한다.

## 3. 검증 루프는 어떻게 만들어야 하나?

### 질문

> 검증 루프는 어떻게 만들어야 하나요?

### 답변 프레임

```text
1. 검증 루프는 AI가 실패를 보고 다시 고칠 수 있게 만드는 장치다.
2. test, typecheck, lint, e2e 같은 자동 검증을 둔다.
3. SPEC 항목별 요구사항 체크를 포함한다.
4. 브라우저, 스크린샷, 실제 플로우 같은 유저 관점 확인도 포함한다.
```

### 말하기용 답변

검증 루프는 AI가 실패를 보고 다시 고칠 수 있게 만드는 장치입니다.

테스트만 말하는 것은 아닙니다.
unit test, integration test, e2e, typecheck, lint처럼 AI가 반복 실행할 수 있는 자동 검증이 기본입니다.

여기에 SPEC 항목별 요구사항 체크가 붙어야 합니다.
그리고 브라우저에서 버튼이 보이는지, 권한별로 다르게 보이는지, 저장 버튼을 눌렀을 때 실제로 저장되는지 같은 유저 관점 확인도 필요합니다.

검증 루프가 없으면 AI는 "완료했다"고 말해도 실제로는 요구사항과 다르게 구현했을 수 있습니다.

### 핵심 한 줄

> 검증 루프는 테스트만이 아니라 자동 검증, 요구사항 체크, 유저 관점 확인을 묶어 AI가 실패를 보고 다시 고치게 만드는 장치다.

## 4. AI가 테스트를 다 통과했다고 하면 믿어도 되나?

### 질문

> AI가 테스트를 다 통과했다고 하면 믿어도 되나요?

### 답변 프레임

```text
1. 테스트 통과는 중요한 신호다.
2. 하지만 테스트 통과가 완료를 의미하지는 않는다.
3. 테스트는 작성된 요구사항과 케이스만 검증한다.
4. 사람의 코드 구조 리뷰와 유저 관점 확인이 필요하다.
```

### 말하기용 답변

테스트 통과는 중요한 신호입니다.
하지만 테스트를 통과했다고 해서 작업이 끝났다고 보면 안 됩니다.

테스트는 작성된 요구사항과 케이스만 검증합니다.
테스트에 없는 암묵지, 사용자 기대, 운영 리스크, UI 연결 문제는 놓칠 수 있습니다.

예를 들어 저장 API 테스트는 통과했지만 실제 UI 버튼과 연결이 안 되어 있을 수 있습니다.
권한 체크 코드는 있지만 화면에서는 잘못 보일 수 있습니다.
저장 완료 메시지는 뜨지만 실제 데이터가 저장되지 않을 수도 있습니다.

그래서 테스트 통과 이후에도 사람이 코드 구조와 영향 범위를 보고, 실제 유저 관점에서 기능이 동작하는지 확인해야 합니다.

### 핵심 한 줄

> 테스트 통과는 완료가 아니라 중요한 검증 신호 중 하나이며, 최종 완료 판단은 코드 구조 리뷰와 유저 관점 확인까지 포함해야 한다.

## 5. AI가 만든 코드는 어떻게 리뷰해야 하나?

### 질문

> AI가 만든 코드는 어떻게 리뷰해야 하나요?

### 답변 프레임

```text
1. 모든 라인을 같은 밀도로 보는 리뷰는 아니다.
2. 구조/의미 단위로 본다.
3. 요구사항 충족, 기존 패턴 적합성, 재사용성, 영향 범위, 운영 리스크를 본다.
4. AI가 성공한 것처럼 보이면서 중복과 기술부채를 쌓지 않았는지 확인한다.
```

### 말하기용 답변

AI가 만든 코드를 리뷰할 때 모든 라인을 같은 밀도로 볼 필요는 없습니다.
하지만 그렇다고 코드를 안 봐도 된다는 뜻은 아닙니다.

사람은 구조와 의미 단위로 봐야 합니다.

먼저 요구사항을 실제로 만족하는지 봅니다.
그 다음 기존 코드 구조와 패턴에 맞는지, 모듈 경계를 흐리지 않는지, 데이터 흐름이나 권한 흐름을 깨지 않는지 확인합니다.

또 기존 함수를 재사용해야 하는데 중복 로직을 새로 만들지는 않았는지, 이 변경이 다른 기능, 운영, 보안, 데이터 정합성, 사용자 경험에 영향을 주지 않는지 봐야 합니다.

AI는 실패해서만 위험한 것이 아닙니다.
성공한 것처럼 보이면서 기존 구조를 무시하고 조용한 기술부채를 쌓을 수 있습니다.

그래서 사람의 리뷰는 줄어드는 것이 아니라, 라인 단위에서 구조/의미/맥락/재사용성/영향 범위 단위로 이동합니다.

### 핵심 한 줄

> AI 코드 리뷰는 모든 라인을 같은 밀도로 보는 것이 아니라, 요구사항 충족, 구조 적합성, 재사용성, 영향 범위, 운영 리스크를 사람이 판단하는 것이다.

## 6. QA로 넘기기 전에 개발자는 무엇을 확인해야 하나?

### 질문

> QA로 넘기기 전에 개발자는 무엇을 확인해야 하나요?

### 답변 프레임

```text
1. QA는 개발자 확인을 대체하지 않는다.
2. AI 검증 루프를 먼저 돌린다.
3. 사람이 코드 구조와 영향 범위를 리뷰한다.
4. 개발 환경에서 요구사항대로 동작하는지 유저 관점으로 확인한다.
```

### 말하기용 답변

QA는 개발자의 확인을 대체하는 단계가 아닙니다.

AI로 개발했더라도 개발자는 QA에 넘기기 전에 자기 기능이 요구사항대로 동작한다고 판단할 수 있어야 합니다.

먼저 테스트, typecheck, lint, e2e, 요구사항 체크 같은 AI 검증 루프를 통과해야 합니다.
그 다음 사람이 코드 구조와 영향 범위를 봐야 합니다.

마지막으로 개발 환경에서 실제 유저 관점으로 확인해야 합니다.
버튼이 보이는지, 권한별 동작이 맞는지, 클릭이 되는지, 저장이 실제로 되는지, UI가 가려져 있지 않은지 확인해야 합니다.

이 상태가 된 뒤에 QA로 넘겨야 합니다.
QA는 개발자가 확인하지 않은 산출물을 대신 완성해주는 단계가 아니라, 3자 관점에서 블랙박스 검증을 하는 단계입니다.

### 핵심 한 줄

> QA로 넘기기 전 개발자는 AI 검증 루프, 코드 구조 리뷰, 개발 환경의 유저 관점 동작 확인까지 끝내야 한다.

## 7. AI가 엉뚱한 방향으로 계속 고치면 어떻게 해야 하나?

### 질문

> AI가 엉뚱한 방향으로 계속 고치면 어떻게 해야 하나요?

### 답변 프레임

```text
1. 고쳐줘 반복을 멈춘다.
2. 원인과 실패 조건을 사람이 다시 좁힌다.
3. SPEC과 검증 기준을 재확인한다.
4. 작업 단위를 줄이거나 사람이 직접 수정한다.
```

### 말하기용 답변

AI가 엉뚱한 방향으로 계속 고치면 "다시 고쳐줘"만 반복하면 안 됩니다.

그 루프가 계속된다는 건 AI가 문제의 원인이나 맥락을 잘못 잡고 있을 가능성이 큽니다.
이때는 사람이 멈추고 원인을 다시 좁혀야 합니다.

먼저 테스트 실패인지, 요구사항 불일치인지, UI 동작 문제인지, 구조적 영향인지 구분해야 합니다.
그 다음 SPEC과 검증 기준을 다시 확인해야 합니다.

필요하면 작업 단위를 더 작게 나눠야 합니다.
그래도 계속 헛돌면 사람이 직접 코드를 수정하거나, 최소한 원인과 수정 방향을 명확히 잡은 뒤 AI에게 그 범위만 맡겨야 합니다.

### 핵심 한 줄

> AI가 계속 엉뚱하게 고치면 고쳐줘 반복을 멈추고, 사람이 원인과 검증 기준을 다시 좁힌 뒤 작은 범위로 맡겨야 한다.

## 8. 작은 수정도 AI에게 맡기는 게 항상 좋은가?

### 질문

> 작은 수정도 AI에게 맡기는 게 항상 좋은가요?

### 답변 프레임

```text
1. 항상 좋은 것은 아니다.
2. AI 위임에도 지시, 대기, 검토, 수정 비용이 있다.
3. 사람이 10초에 고칠 수 있는 일은 직접 하는 편이 나을 수 있다.
4. 위임 여부는 생성 속도뿐 아니라 검토 비용까지 포함해서 판단한다.
```

### 말하기용 답변

작은 수정까지 항상 AI에게 맡기는 것이 좋은 것은 아닙니다.

AI에게 일을 맡기는 것에도 오버헤드가 있습니다.
무엇을 고칠지 설명해야 하고, AI가 수정하는 동안 기다려야 하고, 결과를 다시 검토해야 합니다.

사람이 10초면 고칠 수 있는 문구 수정이나 작은 코드 수정은 직접 하는 편이 더 빠를 수 있습니다.
몇 페이지 문서에서 2~3단어만 고치면 되는 상황이라면 AI에게 시키는 것보다 직접 고치는 게 낫습니다.

중요한 건 AI를 많이 쓰는 것이 아니라, 효과적으로 쓰는 것입니다.
반복적이고 검증 가능한 작업, 초안이 필요한 작업, 여러 후보가 필요한 작업은 AI에게 맡기기 좋습니다.

### 핵심 한 줄

> AI 위임에도 지시와 검토 비용이 있으므로, 작은 수정은 직접 고치는 것이 더 빠를 수 있다.

## 9. 바이브코딩과 에이전틱 엔지니어링은 뭐가 다른가?

### 질문

> 바이브코딩과 에이전틱 엔지니어링은 뭐가 다른가요?

### 답변 프레임

```text
1. 바이브코딩은 해줘/고쳐줘 반복에 가깝다.
2. 에이전틱 엔지니어링은 SPEC, 검증 루프, 코드 리뷰, 유저 확인을 갖춘다.
3. 핵심 차이는 사람이 코드를 이해하고 AI를 통제한다는 점이다.
4. 최종 검토자와 책임자는 AI가 아니라 사람이다.
```

### 말하기용 답변

바이브코딩은 AI에게 "해줘", "고쳐줘"를 반복하는 방식에 가깝습니다.
작은 도구나 개인용 헛간을 만들 때는 그 방식도 쓸 수 있습니다.
유튜브를 보고 이것저것 붙여 만든 헛간처럼, 문이 잘 안 닫히면 판자로 막고 물이 조금 새면 고쳐가며 쓸 수 있는 범위라면 가능합니다.

하지만 중요한 시스템에서는 부족합니다.
AI가 만든 코드가 왜 그렇게 되었는지 모르고, 기존 구조와 어떻게 연결되는지 모르고, 최종 동작도 확인하지 않으면 검증되지 않은 복잡도가 쌓입니다.

에이전틱 엔지니어링은 다릅니다.
사람이 요구사항과 성공 기준을 잡고, AI가 처리할 작업을 나누고, 검증 루프를 만들고, AI 산출물을 리뷰하고, 실제 유저 관점에서 동작을 확인합니다.

특히 중요한 건 사람이 코드를 이해하고 있다는 점입니다.
모든 라인을 외우라는 뜻은 아니지만, 구조와 의미와 영향 범위는 이해해야 합니다.

그리고 AI가 일을 하는 동안에도 사람이 AI를 통제하고 있어야 합니다.
AI가 이상한 방향으로 가면 멈추고, 원인을 좁히고, 직접 수정하거나 더 작은 범위로 다시 맡길 수 있어야 합니다.

최종 검토자와 책임자는 AI가 아니라 사람입니다.

### 핵심 한 줄

> 에이전틱 엔지니어링은 AI에게 맡기되, 사람이 코드를 이해하고 AI를 통제하며 최종 검토자로 책임지는 개발 방식이다.

## 10. AI 개발에서 개발자가 최종적으로 책임져야 하는 것은 뭔가?

### 질문

> AI 개발에서 개발자가 최종적으로 책임져야 하는 것은 뭔가요?

### 답변 프레임

```text
1. AI가 했다는 것은 면책 사유가 아니다.
2. 개발자는 요구사항과 성공 기준을 책임진다.
3. 코드 구조, 영향 범위, 검증 루프를 책임진다.
4. 유저 관점 동작과 QA 전달 품질을 책임진다.
```

### 말하기용 답변

AI 개발에서도 최종 책임자는 개발자입니다.

AI가 코드를 작성했거나 테스트를 돌렸다는 것은 면책 사유가 아닙니다.
내가 AI에게 맡겼다면, 그 결과를 이해하고 검토하고 검증할 책임도 나에게 있습니다.

개발자는 요구사항과 성공 기준을 책임져야 합니다.
또 코드 구조, 영향 범위, 검증 루프도 책임져야 합니다.

마지막으로 QA에 넘기는 품질을 책임져야 합니다.
개발 환경에서 내가 만들려던 기능이 실제로 동작한다고 판단한 뒤 QA로 넘겨야 합니다.

AI는 개발자의 일을 돕는 도구이지, 개발자의 책임을 대신 지는 주체가 아닙니다.

### 핵심 한 줄

> AI가 작성했더라도 개발자는 요구사항, 코드 구조, 검증 루프, 유저 관점 동작, QA 전달 품질의 최종 책임자다.
