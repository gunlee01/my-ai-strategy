# 생산성 전환과 10배 생산성 신화

## 핵심 주장

AI는 생산성을 올린다.

하지만 그 의미는 개발자의 일이 사라진다는 뜻이 아니다.

문서와 코드를 직접 치는 시간이 줄어들고, 그 시간은 AI에게 일을 시키고, 산출물을 검토하고, 방향을 교정하고, 요구사항과 검증 기준을 관리하는 시간으로 이동한다.

## 생산성의 위치가 바뀐다

AI 도입 전에는 사람이 직접 작성하는 시간이 컸다.

AI 도입 후에는 직접 작성 시간은 줄어들지만, 검토와 조율의 비중이 커진다.

```text
기존 개발
  = 사람이 문서 작성
  + 사람이 코드 작성
  + 사람이 리뷰 / 테스트 / QA 대응

AI 활용 개발
  = AI에게 작업 지시
  + AI 산출물 검토
  + 방향 교정
  + 검증 루프 관리
  + 사람의 최종 판단
```

따라서 생산성 향상은 "사람이 필요 없어짐"이 아니라, 사람의 시간이 더 높은 수준의 판단과 통제로 이동하는 현상에 가깝다.

## 코드 생산 10배와 전체 개발 10배는 다르다

코드 작성 속도가 10배 빨라져도 전체 개발 속도가 10배 빨라지는 것은 아니다.

전체 개발 라이프 사이클에는 코드 작성 외에도 여러 단계가 있다.

```text
요구사항 이해
  ↓
SPEC 정리
  ↓
구현
  ↓
테스트 / 타입체크 / lint
  ↓
코드 리뷰
  ↓
통합
  ↓
유저 관점 동작 확인
  ↓
QA
  ↓
운영 영향 판단
```

AI가 구현 단계를 빠르게 만들어도, 검토와 검증해야 할 산출물이 늘어난다.

따라서 병목은 코드 작성에서 요구사항, 리뷰, 검증, 통합, 운영 판단으로 이동한다.

## 10배가 가능한 영역과 아닌 영역

어떤 영역에서는 10배에 가까운 개선이 가능하다.

반복 문서 초안, 보일러플레이트, 작은 스크립트, 테스트 초안, 단순 변환 작업은 크게 빨라질 수 있다.

하지만 모든 개발이 그렇게 빨라지는 것은 아니다.

| 영역 | AI 효과 | 이유 |
|---|---|---|
| 반복 문서 / 코드 초안 | 매우 큼 | 목표가 좁고 결과 검토가 쉬움 |
| 보일러플레이트 / 단순 변환 | 큼 | 패턴이 명확함 |
| 테스트 초안 / 케이스 확장 | 큼 | 반복 생성에 강함 |
| 복잡한 요구사항 해석 | 제한적 | 암묵지와 우선순위 판단이 필요 |
| 큰 코드베이스 변경 | 제한적 | 전역 맥락과 영향 범위 판단이 필요 |
| 운영 장애 / 보안 / 데이터 리스크 | 제한적 | 책임 있는 판단과 조직 맥락이 필요 |
| 아주 작은 직접 수정 | 때로는 느림 | 사람이 10초에 고칠 일을 AI가 분 단위로 처리할 수 있음 |

## 하방 상승을 10배로 착각할 수 있다

10배 생산성처럼 보이는 측정이 실제로는 하방 상승일 수 있다.

예를 들어 어떤 사람이 특정 작업을 거의 못해서 생산성이 0.1에 가까웠다면, AI 도움으로 1까지 올라갈 수 있다.

이때 상대 개선율은 매우 크게 보인다.

하지만 이것은 상방이 10배 올라간 것이 아니라, 낮은 출발점이 실무 가능한 수준으로 올라간 것이다.

```text
0.1 -> 1
  = 10배 개선처럼 보임
  = 하방 상승

10 -> 100
  = 상방 10배 상승
  = 중요한 시스템에서는 훨씬 어려움
```

이 구분이 중요하다.

AI는 많은 사람의 하방을 올리지만, 중요한 시스템을 더 높은 품질과 책임 수준으로 운영하는 상방을 자동으로 올리지는 않는다.

## 리더에게 설명할 때의 핵심

리더에게는 AI 생산성을 인원 축소 논리로 바로 연결하면 안 된다고 설명해야 한다.

더 정확한 설명은 아래에 가깝다.

```text
AI는 직접 작성 시간을 줄인다
  ↓
사람의 시간은 지시 / 검토 / 검증 / 구조 판단으로 이동한다
  ↓
같은 인원이 더 큰 복잡도와 더 많은 검증을 다룰 수 있다
  ↓
하지만 전체 개발 속도는 병목 단계에 의해 제한된다
  ↓
따라서 "코드 생성 10배 = 조직 생산성 10배 = 인원 90% 축소"은 성립하지 않는다
```

## 비용 혁신은 결과다

AI로 비용 혁신은 가능하다.

하지만 비용 혁신은 선결론이 아니라 성공적인 AI 활용의 결과다.

먼저 우리 도메인과 업무 특성에서 AI가 어디까지 잘 작동하는지 확인해야 한다.

```text
AI 적용 후보 찾기
  ↓
작은 업무 단위에서 성공 패턴 검증
  ↓
검증 루프와 사람 개입 지점 설계
  ↓
처리량 / 품질 / 리드타임 / 재작업 비용 측정
  ↓
비용 구조 조정 판단
```

비용 절감은 여러 방식으로 나타날 수 있다.

| 방식 | 의미 |
|---|---|
| 처리량 증가 | 같은 인원이 더 많은 요구사항, 실험, 개선을 처리 |
| 반복 작업 감소 | 보일러플레이트, 문서 초안, 테스트 초안, 단순 검증 자동화 |
| 품질 비용 감소 | QA 재작업, 장애 대응, 유지보수 비용 감소 |
| 인력 조정 | 성공적 적용과 측정 이후 가능한 선택지 중 하나 |

중요한 것은 인원 축소를 먼저 정하고 AI를 끼워 맞추는 것이 아니다.

성공적인 AI 활용을 먼저 만들고, 그 결과로 어떤 비용이 실제로 줄어드는지 측정해야 한다.

도메인마다, 업무마다 AI가 잘하는 범위가 다르므로 일반적인 10배 생산성 주장만으로 비용 절감 규모를 확정할 수 없다.

## 성과 측정 프레임

AI 도입 성과는 코드 생성량으로 측정하면 안 된다.

측정해야 할 것은 요구사항을 만족하고 검증을 통과한 변경이 얼마나 빠르고 안정적으로 나오는가다.

```text
코드 생성량
  != 검증된 개발 성과

검증된 개발 성과
  = 요구사항 충족
  + 리뷰 통과
  + 테스트 통과
  + QA 진행성
  + 낮은 재작업률
  + 낮은 회귀 / 장애율
  + 안정적인 배포
```

특히 리더 관점에서는 라이프사이클 전체 지표를 봐야 한다.

| 지표 | 의미 |
|---|---|
| 리드타임 | 요구사항 확정부터 배포 가능한 상태까지 걸리는 시간 |
| QA 반려율 | QA에서 기본 기능 미동작이나 요구사항 누락으로 되돌아오는 비율 |
| 재작업률 | 구현 후 다시 고치는 비율 |
| 회귀/장애율 | 기존 기능이 깨지거나 운영 문제가 생기는 비율 |
| 배포 안정성 | 배포 후 롤백, 핫픽스, 장애 대응이 줄어드는지 |
| 검토 효과 | 리뷰에서 중요한 구조 리스크와 영향 범위를 발견했는지 |

리뷰 시간은 단독 성과 지표로 보기 어렵다.
AI 산출물이 늘고 사람이 더 책임 있게 검토하면 리뷰 시간은 늘어날 수도 있다.
중요한 것은 리뷰 시간이 짧아졌는지가 아니라, 검토를 통해 중요한 리스크를 잡고 QA 반려, 재작업, 회귀를 줄였는지다.

AI가 코드를 많이 만들었는데 QA 반려가 늘고, 재작업과 회귀가 늘면 생산성이 오른 것이 아니다.

반대로 생성량 자체가 크게 보이지 않아도 리드타임이 줄고, QA가 원활해지고, 재작업과 장애가 줄었다면 실질적인 성과다.

따라서 경쟁력을 위해 올려야 하는 것은 코드 생성 속도가 아니라 검증된 변경이 나오는 속도다.

또 업무 유형별로 효과를 따로 봐야 한다.

반복 문서화, 보일러플레이트, 테스트 작성, 단순 수정에서는 효과가 크게 나올 수 있다.
하지만 큰 설계 변경, 운영 리스크가 큰 작업, 복잡한 요구사항 해석에서는 다른 결과가 나올 수 있다.

## 한 문장 요약

> AI 생산성 향상은 사람이 사라지는 것이 아니라, 사람이 직접 쓰는 시간이 AI 산출물을 지시하고 검토하고 검증하는 시간으로 이동하는 것이다.
