# AI 위임 기준

## 핵심 주장

AI에게 과감히 맡길 일과 사람이 반드시 봐야 할 일은 구분해야 한다.

기준은 검증 가능성, 실패 비용, 맥락 의존성, 책임 규모다.

다만 이 기준은 "AI에게 맡길 수 있는 일"과 "절대 맡기면 안 되는 일"을 정적으로 나누는 표가 아니다.

사람이 검증한 좋은 계획과 잘게 분할된 작업을 주면 AI는 곧잘한다.
AI가 좁은 맥락만 보는 약점도 좋은 계획, 명확한 성공 기준, 작은 작업 단위, 검증 루프로 어느 정도 극복할 수 있다.

하지만 그래도 통제를 벗어나 목표와 다른 방향으로 그럴듯하게 진행하거나, 우리가 원하는 수준의 결과가 나오지 않는 경우가 많다.
그래서 휴먼 인 더 루프는 필수다.
사람은 계획을 검증하고, 작업을 나누고, 중간 결과를 보고 방향을 교정하고, 최종 산출물을 리뷰해야 한다.

## AI에게 맡기기 좋은 일

AI에게 맡기기 좋은 일은 대체로 아래 조건을 만족한다.

```text
목표가 좁다
  + 결과를 검증할 수 있다
  + 실패 비용이 낮다
  + 반복 가능하다
  + 되돌리기 쉽다
```

예시는 다음과 같다.

- 문서 초안
- 테스트 초안
- 보일러플레이트
- 단순 변환
- 작은 스크립트
- 국지적 리팩터링 초안
- 반복 검증 실행
- 변경점 요약

이런 작업은 AI가 빠르게 초안을 만들고, 자동 검증 루프나 사람 검토로 결과를 확인하기 쉽다.

중요한 것은 AI에게 맡긴 뒤 방치하지 않는 것이다.
사람이 중간 결과를 보고 목표와 어긋나면 즉시 방향을 다시 잡아야 한다.

## 사람이 봐야 하는 일

사람이 반드시 봐야 하는 일은 아래 조건을 가진다.

```text
맥락 의존성이 높다
  + 실패 비용이 크다
  + 책임 규모가 크다
  + 되돌리기 어렵다
  + 암묵지와 트레이드오프가 많다
```

예시는 다음과 같다.

- 요구사항 해석
- SPEC 최종 확정
- 모듈 경계와 구조 판단
- 영향 범위 판단
- 운영 리스크 판단
- 보안/권한/데이터 관련 결정
- 사용자 경험 최종 확인
- QA로 넘기기 전 개발자 확인
- 장애 원인 좁히기와 대응 방향 결정

AI가 초안을 만들거나 후보를 제시할 수는 있다.

하지만 최종 판단은 사람이 해야 한다.

## 네 가지 기준

| 기준 | AI에게 맡기기 좋은 쪽 | 사람이 봐야 하는 쪽 |
|---|---|---|
| 검증 가능성 | 테스트, 실행, 화면, diff로 확인 가능 | 정답이 모호하고 판단 기준이 암묵적 |
| 실패 비용 | 잘못되어도 쉽게 고치거나 되돌릴 수 있음 | 장애, 보안, 데이터, 고객 피해가 큼 |
| 맥락 의존성 | 필요한 맥락이 작고 명시적 | 도메인, 조직, 운영, 과거 결정이 중요 |
| 책임 규모 | 개인용 도구, 작은 내부 작업 | 많은 사람이 쓰는 운영 시스템 |

## 판단 흐름

```text
이 작업의 목표가 좁은가?
  ├─ 아니오 -> 사람이 먼저 구조화
  └─ 예
      ↓
결과를 자동/수동으로 명확히 검증할 수 있는가?
  ├─ 아니오 -> 사람 검토 지점 강화
  └─ 예
      ↓
실패 비용이 낮고 되돌리기 쉬운가?
  ├─ 아니오 -> 사람 승인 후 진행
  └─ 예
      ↓
사람이 검증한 계획으로 작업을 쪼갰는가?
  ├─ 아니오 -> 사람이 먼저 계획과 작업 단위를 조정
  └─ 예
      ↓
AI에게 과감히 위임하고 검증 루프를 돌린다
      ↓
중간 결과가 목표에 맞는가?
  ├─ 아니오 -> 사람이 방향을 교정하고 작업 단위를 다시 조정
  └─ 예
      ↓
최종 결과를 사람이 리뷰하고 확정한다
```

## 한 문장 요약

> AI는 좋은 계획과 분할된 작업을 주면 잘하지만, 사람이 중간 결과와 최종 결과를 목표에 맞게 계속 통제해야 한다.
